# RentWise v0 вҖ” Implementation Design Spec

**Status:** Live В· Build: scaffold-v0 В· Date: 2026-05-01
**Scope:** Multi-agent rental advisory chat with dynamic re-ranking, deployed as a runnable local prototype against a 1,087-listing Bay Area dataset.

---

## 1. v0 Scope Definition

### In Scope (Live in v0)
- Multi-agent group chat with **4 specialist agents** + **LLM-driven router**
- Conversational discovery: any agent may ask clarifying questions when blocked
- Dynamic user profile extraction with add / remove / negation operations
- Heuristic ranking with explicit per-feature score breakdown
- Live shortlist auto-pinning + dynamic re-ranking on every preference change
- User-editable profile chips with surgical removal
- Cross-source data merging (Zillow + Craigslist) with spatial deduplication
- @-mention typeahead for explicit agent dispatch
- Light-theme consulting-grade UI (chat + sidebar + live shortlist rail)

### Stubbed (UX visible, integration deferred to v1)
- **Real-time commute calculation** вҖ” currently Haversine straight-line distance only; Google Maps Distance Matrix is the v1 dependency.
- **Outreach send/receive** вҖ” emails are drafted and rendered as preview cards; Gmail OAuth + send + reply ingestion deferred to v1.

### Out of Scope (Future)
- Trained ML rankers (LTR / Mixture of Experts / two-tower) вҖ” gated on accumulating 5kвҖ“10k engagement events.
- CV-based aesthetic scoring on listing photos.
- Authentication, multi-user state, billing.
- Native mobile clients.

## 3. Specialist Agent Specification

| Agent | Role | Primary Tool Surface |
|---|---|---|
| **Search** | NL preferences вҶ’ ranked candidates | Profile-based hard filter + RankingService + LLM rerank top 25 |
| **Property Analyst** | Q&A over listing facts | Reads canonical fields; cites or admits "not listed in source" |
| **Location & Commute** | Geographic context | Walk/transit/bike scores В· schools В· neighborhood text mining В· Haversine distances |
| **Outreach** | Inquiry email drafting | Builds CAN-SPAM-compliant drafts using user profile + listing context |

**Agent Router (LLM-driven dispatch):** A central Claude Sonnet call inspects each user message + chat context and selects the responding agent. Explicit `@search` / `@property` / `@location` / `@outreach` mentions override the router deterministically.

**Multi-turn Multi-agent Dialogue:** When an agent cannot proceed (missing budget, missing destination address, missing user name), it returns a clarifying question and stamps `session.pending_clarification = (agent_id, [needed_fields])`. The router uses this as a strong routing hint so the user's reply returns to the asking agent.

## 4. User Profile & Dynamic Re-Ranking

### Profile Schema
```
budget_max В· beds_min В· beds_max В· pets[] В· must_haves[] В·
nice_to_haves[] В· avoid[] В· neighborhoods[] В· commute{name, lat, lng}
user_name В· move_in_date В· notes
```

### Profile Update Engine
A lightweight LLM call (Claude Sonnet) runs after every user turn and emits a JSON patch with **add**, **remove**, and **clear** operations:

- **Add ops:** `must_haves_add`, `pets_add`, `neighborhoods_add`, etc.
- **Remove ops:** `must_haves_remove`, `nice_to_haves_remove`, `avoid_remove`, etc.
- **Clear ops:** `clear_commute`, `clear_budget`, `clear_beds`

Negation handling: when the user says *"I don't want X"* / *"дёҚиҰҒ X"*, the updater both **removes related stale entries** from any list AND **adds X to `avoid`** вҖ” preventing chip-state contradictions.

User can also surgically delete any inferred fact via the `вң•` button on each profile chip in the left sidebar.

### Heuristic Ranking Service
Component-weighted scoring; each feature contributes 0вҖ“10 with weights below. Inactive features (not in profile) are skipped, not penalized.

| Component | Weight | Logic |
|---|---:|---|
| budget | 3.0 | piecewise: 10 if rent вүӨ 85% of cap; scales 5вҖ“10 between 85вҖ“100%; penalty above |
| beds | 2.0 | 10 if any available bed-count matches request; degrades by distance |
| pets | 2.0 | 10 if all requested pet types allowed |
| must_haves | 3.0 | % of must-have keywords matched in description/amenities |
| nice_to_haves | 1.0 | % of nice-to-have keywords matched |
| avoid | 1.5 | penalty per avoided keyword found |
| commute | 2.5 | banded by Haversine miles to commute target |
| walk_score | 1.0 | walk_score / 10 |
| transit_score | 0.5 | transit_score / 10 |
| neighborhood | 1.5 | 10 on substring match |

**Lifecycle:** every user turn re-evaluates the profile, then `rescore_shortlist()` re-runs the scoring against every shortlisted listing and re-orders the right rail. Display order = display order; ordinal references ("the second one") resolve correctly.

## 6. Architectural Rationale: Heuristic vs. Two-Tower

The two-tower (DSSM-style) architecture represents the optimal configuration for a production-grade recommendation engine, mirroring the Airbnb research cited in our technical specifications. Pre-computed item embeddings + real-time query encoding + ANN lookup yields O(log N) inference scaling to millions of entities.

**Strategic Assessment: Opposing Immediate Refactoring**

1. **Data Scarcity (Zero Training Corpus).** A two-tower's primary utility is its capacity to learn latent preferences from `(query, listing, engagement)` triples. Current metrics: 0 active users, 0 clickstream logs, 0 conversion events. In a zero-data environment, a two-tower implementation collapses into dual encoders + cosine similarity вҖ” essentially BM25-class text retrieval. It will not outperform our heuristic during cold-start and risks ignoring hard constraints (budget, pets, bedrooms) that drive tenant decisions.

2. **Interpretability Thresholds.** Our breakdown (`budget: 10/10, commute: 10/10, must_haves: 10/10`) is functionally incompatible with raw two-tower output, which yields a single dot product (`cosine: 0.87`) вҖ” obscuring the *why* behind the ranking and breaking the user-surface transparency requirement.

3. **MVP Cold-Start Economics.** Implementing a complex reranker now would yield lower-quality results for the first 1,000 users вҖ” sacrificing early-adopter sentiment for structural sophistication.

**Architectural Roadmap: The "v1.5 Semantic Blend"**

```
Stage 1 (v0, current)     Hard filter + heuristic score, ~3ms
        вҶ“ top 50
Stage 1.5 (immediate)     + sentence-transformer cosine as one component
        вҶ“ top 5
Stage 2 (Month 4вҖ“6)       Trained two-tower reranker, on real engagement data
```

**v1.5 Immediate Benefits**
- Captures qualitative intent ("vintage character", "modern luxury") via description encoding.
- Preserves explainability: `semantic_match` becomes one row in the breakdown.
- Forward-compatible: embeddings generated now serve as the future item-tower output.

**Execution Requirements for Two-Tower Viability**
1. Instrument PostHog events: `listing_viewed`, `listing_pinned`, `outreach_sent`, `tour_scheduled`.
2. Log every `(query, top-5 returned, engagement signal)` triple.
3. Wait for 5kвҖ“10k sessions before freezing the reranker architecture.

## 7. Data Ingestion Pipeline

### Final System State вҖ” `listings_loaded: 1,087`

| Data Source | Count |
|---|---:|
| Zillow apartment_building (San Jose) | 256 |
| Zillow single_home_raw, normalized (San Jose) | 620 |
| Craigslist new вҖ” multi-city Bay Area | 209 |
| Both Zillow + Craigslist (merged record) | 2 |

**Regional Profile (Craigslist 209):** San Jose (23), Oakland (22), San Francisco (20), Hayward (14), Fremont (13), Sunnyvale (7), Santa Cruz (6), Santa Rosa (5), Berkeley (5), Richmond (4); 1вҖ“3 listings each in Daly City, Vallejo, Napa, Pittsburg, Redwood City, San Mateo, Campbell, Walnut Creek, Palo Alto, Milpitas, Mountain View, El Cerrito; 31 records remain in a generic "Bay Area" bucket for edge locations.

**Pipeline Stages (`load_listings()` logic)**

1. **Enriched Source Loading** вҖ” ingests 910 raw Zillow JSONL records.
2. **Normalization & Cleansing** вҖ” eliminates empty-shell records and re-shapes 620 `single_home_raw` records into apartment-style schema by mapping `raw.property.*` and `raw.property.resoFacts.*` to canonical top-level fields (878 cleaned).
3. **Multi-Channel Integration** вҖ” processes 352 Craigslist CSV rows through the normalization engine; each gets a synthetic `zpid: "cl_<post_id>"` and city tag derived from lat/lng bounding-box lookup.
4. **Spatial Deduplication** вҖ” Haversine threshold вүӨ 50m + bed-count overlap check identifies duplicate entities across sources.
5. **Intelligent Record Merging** вҖ” Craigslist data resolves Zillow gaps (lease terms, pet policies, additional floor plans, descriptions, deposits); Zillow's structured fields take precedence; merged records carry `_data_sources: ["apartment_building", "craigslist"]` and a `_craigslist_post_ids` array.
6. **Listing Augmentation** вҖ” pure-Craigslist records are tagged with city metadata via coordinate-based lookup over a 30-bbox table covering South Bay, East Bay, Peninsula, San Francisco, Marin, North Bay, and overflow zones (Santa Cruz, Sacramento).

**End-to-End Search Validation (Smoke Test)**
- `"1BR in Oakland under $2500"` вҶ’ Oakland Craigslist listings, scoring range 82вҖ“100 вң…
- `"studio in San Francisco"` вҶ’ SF Craigslist inventory вң…
- `"3BR family home under $5500, dog-friendly, near good schools"` вҶ’ mixed apartment + single-home results вң…
